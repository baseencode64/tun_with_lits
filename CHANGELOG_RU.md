## [1.6.0] - 2026-05-25

### Добавлено

- **Connection persistence & auto-reconnect** — новый механизм автоматического переподключения при исчерпании всех серверов в списке:
  - `Reconnector` — управление переподключением с exponential backoff (5s → 10s → 20s → ... → 5m)
  - Jitter (±25%) для распределения нагрузки при параллельных клиентах
  - Callback для обновления списка серверов перед каждой попыткой
  - Graceful stop через контекст или вызов `Stop()`
  - Подробное логирование каждой попытки переподключения

- **Новые CLI флаги реконнекта:**
  - `--max-retries <n>` — макс. количество попыток (0 = безлимитно, по умолч.: 0)
  - `--min-backoff <duration>` — начальная задержка (по умолч.: 5s)
  - `--max-backoff <duration>` — максимальная задержка (по умолч.: 5m)
  - `--backoff-factor <factor>` — множитель экспоненты (по умолч.: 2.0)

- **Новая секция YAML конфигурации:**

  ```yaml
  reconnection:
    max_retries: 0
    min_backoff: "5s"
    max_backoff: "5m"
    backoff_factor: 2.0
  ```

- **Новый публичный API в пакете `pkg/client`:**
  - `reconnector.go` — `Reconnector`, `ReconnectionConfig`, `ErrReconnectionStopped`
  - `NewReconnector(logger, cfg, reconnectFunc, refreshFunc)` — создание реконнектора
  - `NewVPNConnectorWithReconnect(client, selector, logger, reconnCfg)` — коннектор с кастомным реконнектом
  - `ErrAllServersExhausted` — sentinel error для определения ситуации "все серверы перепробованы"

- **Тесты:** 12 тестов для Reconnector (backoff вычисления, retry до успеха, max retries exceeded, context cancellation, Stop, refresh callback, unlimited retries, reset, defaults)

## [1.5.14] - 2026-05-25

### Исправлено

- **Устранено дублирование DNS маршрутов, вызывавшее предупреждения 'file exists'**
  - Удалены добавления DNS маршрутов из `setupTunnel()` для предотвращения конфликтов с `setupDNSProtection()`
  - DNS маршруты теперь добавляются только один раз через `setupDNSProtection()` когда `enable_dns_protection` включён
  - Исправлено дублирование IPv6 DNS маршрутов в `setupTunnel()`
  - Чистый вывод логов без ошибок 'file exists' при установке соединения

- **Исправлено обновление Prometheus метрик для трафика и длительности соединения**
  - Исправлено преждевременное завершение горутины `startMetricsUpdate()` когда VPN не подключён
  - Горутина теперь ожидает подключения вместо выхода с `return`
  - Метрики `vpn_bytes_read_total`, `vpn_bytes_written_total` и `vpn_connection_duration_seconds` теперь обновляются корректно
  - Исправлен `monitorTraffic()` для постоянного использования `/proc/net/dev` как надёжного источника статистики трафика
  - Удалён ненадёжный wrapper `readerMetrics`, некорректно работавший с TUN интерфейсами
  - Исправлен `readInterfaceStats()` для корректного парсинга имени интерфейса и статистики из `/proc/net/dev`
  - Исправлены индексы полей для парсинга RX/TX байт (RX: индекс 0, TX: индекс 8)

### Изменено

- Упрощена логика DNS маршрутизации — маршруты добавляются только при явном запросе
- Упрощён мониторинг трафика — прямой парсинг `/proc/net/dev` вместо обёртки
- Улучшена поддерживаемость кода за счёт удаления избыточного управления маршрутами

## [1.5.13] - 2026-05-25

### Исправлено

- **Критическая проблема DNS разрешения из-за некорректной реализации защиты от утечек DNS**
  - Исправлена дублирующая настройка DNS защиты, блокировавшая DNS трафик даже без флага `--dns-protection`
  - Удалены некорректные правила iptables REDIRECT, перенаправлявшие DNS трафик на порт SOCKS прокси
  - SOCKS прокси не может обрабатывать сырой DNS протокол, что вызывало полный отказ DNS разрешения
  - DNS теперь корректно маршрутизируется через TUN интерфейс только через записи таблицы маршрутизации
  - Удалены неиспользуемые функции: `setupDNSLeakProtection()`, `blockDirectDNSAccess()`, `addFirewallRule()`, `addFirewallRuleNftables()`, `routeDNSThroughTUN()`
  - `setupDNSTrafficForcing()` теперь корректно ничего не делает, так как DNS маршрутизация обрабатывается существующими маршрутами

### Изменено

- Упрощена защита от утечек DNS — используются только записи таблицы маршрутизации (без манипуляций iptables)
- DNS защита теперь включается только при явном запросе через флаг `--dns-protection`
- Улучшена поддерживаемость кода за счёт удаления мёртвого кода и дублирующего функционала

## [1.4.4] - 2026-05-22

## [1.4.0]

### Добавлено

- **Логирование статуса подключения с IP информацией**
  - Новый метод `GetConnectionStatus()` возвращает детальную информацию о подключении
  - Новый `LogConnectionStatus()` логирует текущий статус каждые 30 секунд
  - Логи включают: TUN интерфейс, локальные IPv4/IPv6 адреса, XRay сервер, статистику трафика
  - Начальный статус логируется сразу после успешного подключения
  - Периодические обновления статуса здоровья и подключения в цикле мониторинга
- **Поддержка множественных raw URL с fallback**
  - Новая опция конфигурации `connection.from_raw_urls` (массив URL)
  - Автоматический fallback: если первый URL не работает, пробуются следующие по порядку
  - Работает как с конфигурационными файлами, так и с CLI (`--from-raw` использует первый успешный URL для обновления)
  - Улучшенная отчетность об ошибках показывает какой URL успешен/неуспешен
  - Обратная совместимость: одиночный `from_raw` продолжает работать
- **Поддержка конфигурационных файлов** (YAML формат)
  - Новый флаг `--config` для загрузки настроек из YAML файла
  - Поддержка всех настроек приложения: подключение, логирование, health monitoring, выбор сервера
  - CLI аргументы переопределяют значения из конфигурационного файла (приоритет: CLI > конфиг > значения по умолчанию)
  - Автоматическая валидация с понятными сообщениями об ошибках
  - Пример конфигурации: `config.yaml.example`
- Структурированное логирование серверов в JSON-совместимом формате
  - Новый метод `LogServerReport()` для структурированного логирования серверов
  - Каждый сервер логируется отдельно с полями: rank, host, port, latency_ms, status
  - Совместимо с ELK Stack, Grafana Loki, Splunk
  - Многострочные текстовые отчёты заменены на машиночитаемые JSON записи

### Исправлено

- **Config file не загружал `from_raw_urls`**: Исправлена проблема, когда множественные raw URL из YAML конфига игнорировались
  - Теперь корректно читает и использует все URL из массива `connection.from_raw_urls`
  - Реализует автоматический fallback между URL при ошибке загрузки

### Изменено

- IPv6 адрес изменён с невалидного `fd00:goxray::1` на корректный ULA `fd00:dead:beef::1`
- Логирование отчётов серверов теперь использует структурированный формат вместо конкатенации строк
- Улучшено логирование периодического обновления с лучшим контекстом ошибок

## Примечание по безопасности

- Всегда проверяйте подлинность сервера перед подключением
- Используйте `TLSAllowInsecure: false` в продакшене (по умолчанию)
- Регулярно обновляйте списки серверов
- Следите за статусом здоровья для аномалий
